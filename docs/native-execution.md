# Mondrian native execution — reference

When does an MDX query get evaluated by **SQL pushed to the warehouse**, and
when does Mondrian load member sets into memory and **interpret** the function
in Java? This doc answers that for the four native paths in `mondrian.rolap.*`
and points at the constraint classes that actually emit the SQL.

Audience: people writing MDX generators (AI-query converter, hand-authored
schemas) and Mondrian developers debugging "why didn't my query go native?"
fallbacks.

> See also: the Calcite-backend rewrite — once that's enabled
> (`-Dmondrian.backend=calcite`), this entire path runs through a different
> SQL emitter. Native *triggers* are unchanged; only the resulting SQL string
> changes. This doc is about the trigger surface, not the emitter.

---

## TL;DR

| MDX shape | Native? | Property to turn off |
| --- | --- | --- |
| `NonEmptyCrossJoin(S1, S2)` | yes | `mondrian.native.crossjoin.enable` |
| `CrossJoin(S1, S2)` under `NON EMPTY` | yes | `mondrian.native.crossjoin.enable` |
| `Filter(S, expr)` | yes | `mondrian.native.filter.enable` |
| `TopCount(S, n, key)` / `BottomCount(...)` | yes | `mondrian.native.topcount.enable` |
| `NON EMPTY level.Members` / `NON EMPTY member.Children` | yes | `mondrian.native.nonempty.enable` |
| `Hierarchize(...)` | **no** | (interpreted) |
| `Order(...)` (without limit) | **no** | (interpreted) |
| `Tail(...)` / `Head(...)` | **no** | (interpreted) |
| Any of the above with a calculated member in args | **no** (auto-fallback) | n/a |
| Any of the above on a ragged or parent-child hierarchy | partially native; post-filtered in Java | n/a |

Native means **one SQL round-trip** with the constraint embedded in
WHERE / HAVING / ORDER BY / LIMIT. Interpreted means **read all candidate
members into memory** and apply the function row-by-row in Java. The
difference is often 100×–1000× on warehouses where the result set is small
relative to the candidate set.

## The four natives

All four extend `RolapNativeSet` and register with `RolapNativeRegistry`. The
registry is **first-match-wins**, ordered: `NonEmptyCrossJoin` → `CrossJoin` →
`TopCount` → `Filter`.

### RolapNativeCrossJoin

**Matches:**

```mdx
NonEmptyCrossJoin([Customer].[Gender].Members, [Product].[Family].Members)
CrossJoin([Customer].[Gender].Members, [Product].[Family].Members)   -- NON EMPTY only
```

**Engages when:**

- evaluator is in non-empty mode (`evaluator.isNonEmpty()`), AND
- all arguments resolve to level-members or explicit member sets (no `ALL`,
  no calc members), AND
- at least one argument has real members, AND
- `SqlContextConstraint.checkValidContext()` finds a fact-table join path.

**Bails out (→ interpreter) when:**

- not in non-empty mode (bare `CrossJoin` outside `NON EMPTY` is always
  interpreted), or
- any argument contains a calculated member
  (`SqlConstraintUtils.containsCalculatedMember`), or
- ragged or parent-child hierarchies — the SQL would over-count; Java
  post-filters the result.

**Disable:** `mondrian.native.crossjoin.enable=false`

### RolapNativeFilter

**Matches:**

```mdx
Filter([Customer].[State].Members, [Measures].[Unit Sales] > 1000)
```

**Engages when:**

- the set argument translates to a valid level/hierarchy crossjoin args, AND
- the filter expression compiles to SQL via `RolapNativeSql` (measures,
  numeric / boolean comparisons, basic arithmetic), AND
- the evaluator context has no calculated members, AND
- valid fact-table join path.

**Bails out when:**

- filter expression references a custom UDF or a non-rewritable measure
  expression, or
- evaluator context contains calculated members.

**Disable:** `mondrian.native.filter.enable=false`

The SQL lands as a `HAVING` clause, generated by
`RolapNativeSql.generateFilterCondition()`. If your warehouse doesn't like
the `HAVING` clause Mondrian emitted, this is the first place to look.

### RolapNativeTopCount

**Matches:**

```mdx
TopCount([Product].[Product Family].Members, 3, [Measures].[Store Sales])
BottomCount([Product].[Product Family].Members, 3, [Measures].[Store Sales])
```

**Engages when:**

- exactly 2 or 3 arguments, AND
- the count is a **literal integer** (not a calculated expression), AND
- the order-by expression (if any) compiles to SQL.

**Bails out when:**

- count is non-literal (e.g. `TopCount(set, [Measures].[N], ...)`), or
- order-by references something not rewritable to SQL.

**Disable:** `mondrian.native.topcount.enable=false`

The SQL lands as `ORDER BY ... LIMIT n` (or the dialect equivalent), via
`RolapNativeSql.generateTopCountOrderBy()` + `TopCountConstraint`.

### NON EMPTY (not a separate native — a mode)

`NON EMPTY level.Members`, `NON EMPTY member.Children`, and
`NON EMPTY member.Descendants` all engage native processing **as a side
effect of the evaluator's non-empty flag** — there's no `RolapNativeNonEmpty`
class. Instead, `evaluator.setNonEmpty(true)` causes `RolapNativeCrossJoin`
(and friends) to treat their arguments as needing a fact-table join.

The constraint that emits the SQL is `NonEmptyCrossJoinConstraint`. It
always forces a join to the fact table so the SQL filters out members with
no aggregable rows.

**Disable:** `mondrian.native.nonempty.enable=false`

## What turns native off (every native)

These conditions cause **all** natives to bail to the interpreter, in addition
to the per-native conditions above:

1. **Calculated members in the evaluator context.** Native SQL can't compute
   a `MEMBER` expression. Detected via
   `SqlConstraintUtils.containsCalculatedMember()`.
2. **`SqlContextConstraint.checkValidContext()` returns false.** Usually
   means a non-conforming dimension or no valid join path to the fact table.
3. **`evaluator.isPreferInterpreter()`** is true. Some compiler paths set
   this when they detect a shape that can't safely round-trip.
4. **Ragged or parent-child hierarchies** in the result tuples — partially
   native; Java post-filters inaccessible / null-parent tuples. Look for
   `filterInaccessibleTuples()` calls.
5. **Custom role-based access control** in effect — same post-filter pattern.

If a query you expected to be native is interpreted, that list is the
checklist. Then turn on `mondrian.native.unsupported.alert=WARN` and look
for the bail-out reason in the log.

## Mondrian properties

Set in `mondrian.properties` or via `-D...` JVM args.

| Property | Default | What it disables |
| --- | --- | --- |
| `mondrian.native.crossjoin.enable` | `true` | `NonEmptyCrossJoin`, `CrossJoin` under `NON EMPTY` |
| `mondrian.native.filter.enable` | `true` | `Filter(...)` |
| `mondrian.native.topcount.enable` | `true` | `TopCount`, `BottomCount` |
| `mondrian.native.nonempty.enable` | `true` | `NON EMPTY level.Members`, `member.Children`, `member.Descendants` |
| `mondrian.native.ExpandNonNative` | `false` | When `true`, expands nested natives even if one child fails — useful for debugging when one bail-out is killing the whole chain. |
| `mondrian.native.unsupported.alert` | `OFF` | `WARN` or `ERROR` to surface bail-outs in the log. |
| `mondrian.native.NativizeMinThreshold` | `100000` | Below this candidate-set cardinality, the nativizer keeps the query interpreted (native overhead not worth it). |
| `mondrian.native.NativizeMaxResults` | `150000` | Above this expected result-set cardinality, bail to interpreter. |

The cardinality thresholds (`NativizeMinThreshold`, `NativizeMaxResults`) are
the ones most worth tuning for production. Defaults are conservative.

## How the AI-query converter's emit maps to native

The AI-query converter (`saiku-core/saiku-service/.../AiSchemaConverter.java`)
emits MDX in patterns that align deliberately with the native-trigger
surface above. Rough mapping:

| Converter input | Emitted MDX | Native? |
| --- | --- | --- |
| `measures + rows` with `nonEmpty: true` (default) | `NON EMPTY level.Members` | yes (NON EMPTY mode) |
| `measures + rows[]` (multiple distinct hierarchies) | `NON EMPTY CROSSJOIN(...)` | yes (RolapNativeCrossJoin) |
| `measures + rows + order + limit` | `NON EMPTY TopCount(level.Members, n, [Measures].[X])` | yes (RolapNativeTopCount) |
| `filters: [...]` (in WHERE) | `WHERE ({m1, m2, ...})` | yes (just slicer; no native needed) |
| `filters: [...]` with `op: "relative"` `last_n_days` | `WHERE Tail(level.Members, n)` | **no** — `Tail` is interpreted; the slicer result set is usually tiny enough not to matter |
| Two rows entries from the **same** hierarchy | `Hierarchize({set1, set2})` | **no** — `Hierarchize` is interpreted |

So the converter's default emit (measures + crossjoined rows + optional
top-N) hits **three** native paths and is generally a single SQL round-trip
on conformant fact tables.

The `Tail` / `Hierarchize` / `Ytd` / `Mtd` / `Qtd` emits are interpreted, by
design — they operate on small dimension-level result sets that don't need
pushdown.

## Debugging: did my query go native?

1. **Log4j logger `mondrian.rolap.RolapNativeSet`** at `DEBUG`: emits the
   generated SQL when a native engages. Most direct signal.
2. **`mondrian.native.unsupported.alert=WARN`**: surfaces the reason for
   bail-outs. The log message names the trigger class and the failed check.
3. **Look at the warehouse query log**: a native query lands as a single
   `SELECT ... GROUP BY ... HAVING ... LIMIT`. An interpreted query lands as
   many small queries (one per level scan).

If you see the small-queries pattern when you expected one big one, work the
"What turns native off" checklist above.

## Source pointers

For source-divers, the bones of native execution:

| Class | Role |
| --- | --- |
| `mondrian.rolap.RolapNative` | base class, listener lifecycle |
| `mondrian.rolap.RolapNativeRegistry` | first-match-wins dispatcher |
| `mondrian.rolap.RolapNativeSet` | shared SQL execution, caching, access-control post-filter |
| `mondrian.rolap.RolapNativeSql` | MDX-expression → SQL-fragment compiler |
| `mondrian.rolap.sql.SqlContextConstraint` | shared join-validation logic |
| `mondrian.rolap.sql.SqlTupleReader` | the SQL workhorse; reads members/tuples from the result set |
| `mondrian.rolap.NonEmptyCrossJoinConstraint` | adds the fact-table join |
| `mondrian.rolap.FilterConstraint` | emits the `HAVING` clause |
| `mondrian.rolap.TopCountConstraint` | emits `ORDER BY ... LIMIT` |

The constraint classes are where bugs usually live when "my query went
native but the SQL is wrong." Start there.
